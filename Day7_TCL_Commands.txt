TCL(Transaction Control Language)
---------------------------------
--> The process of combining all related operations into a single unit and executing them according to the rule "Either all operation should be succeed or none should be succeed". is called transaction management.

***Transaction: A transaction is considered as a single unit of work and follows the principle
"Either all or non"
Example: Fund Transfer.

Consider a money transfer process between two accounts.

1. Debit amount from the sender's account.
2. credit amount into the receiver's account.

Both operations should be executed as a single transaction.

If the amount is debited from the sender's account successfully. But crediting the receiver's account fails, It may lead to the data inconsistency problems.


There are two types of Transactions such as 
1. Local Transactions
--> In this local transaction, all operations of a transaction are executed in the same database
Ex: Funds transfer from one account to anther account. where, both accounts belongs to the same account.

2. Global Transactions
--> In this global transactions, Operations of a transaction are executed on different databases.
Ex: Funds transfer from one account to anther account. where, both accounts belongs to the different account.



ACID Properties
---------------
1. Atomicity: All or Nothing
--> Atomicity ensures that a transaction is completed entirely or not at all. If any operation fails, the entire transaction is rolled back.
2. Consistency: The Database remains in a valid state
--> Consistency ensures that the database remains in a valid state before and after a transaction.
3. Isolation: Transaction do not interface with each other
--> Isolation ensures that multiple transactions running at the same time do not interfere with each other.
4. Durability: Committed changes are permanent
--> Once a transaction is committed, the changes are permanent, even if the system crashes.

***What is locking
   ---------------
Locking is a mechanism used by the database to control concurrent access data

It ensures that multiple users access the same data simultaneously the database maintains consistency, integrity, and prevent conflicts.

1. Shared lock
--> It is used to allow a transaction to read data.
Multiple transactions can hold a shared lock on the same row.
No transaction can modify the row until all shared locks are released.
**Characteristics
- Read only
- Multiple users can read simultaneously
- Blocks updates and deletes

2. Exclusive lock
--> It allows transaction to modify data.
while exclusive lock exists:
No other transaction can tread using a locking read or modify that row
Only the transaction holding the lock can update/delete the data
**Characteristics
- Used for INSERT, UPDATE, DELETE
- Only one exclusive lock at a time
- Blocks both reads (locking reads) and writes


3. Intent lock
--> It is a table-level lock that indicates a transaction intends to place row level locks.
It helps SQL quickly determine whether a table-level lock conflicts with existing row locks.
Users do not usually create Intent locks manually. InnoDB manages them automatically.


Real-Life example: 
--> Imagine a library:
Shared lock: Many students can read the same book at the same time.
Exclusive lock: One student borrows the book, so nobody else can read or borrow it until it is returned.
Intent lock: The librarian marks that some is about to read or borrow books from a shelf, helping coordinate access without checking every book individually.
